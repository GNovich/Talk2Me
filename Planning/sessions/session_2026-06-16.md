## Feature 17 — Latency reduction and output fidelity

**What was done (b, c, d):**

**(b) Thread overlap — question selection during ref_buffer ops:**
- `run_loop()` now submits `engine.next_question()` to a `ThreadPoolExecutor(max_workers=1)` immediately after transcription completes
- Ref buffer push + best_reference() run on the main thread concurrently
- `question_future.result()` collected before TTS; when LLM adapter is on, this overlaps ~1–2 s of personalization with the (fast) ref_buffer operations
- When LLM is off, overhead is negligible (both complete in <5 ms)

**(c) Dual Whisper model — fast model for calibration:**
- `config/exhibit.yaml` adds `stt.model_fast: null` (null = use full model for all turns; set to e.g. `"mlx-community/whisper-small-mlx"` to enable)
- `run_loop()` creates `transcriber_fast` when `model_fast != full_model`
- Calibration turns (turn index ≤ `engine.calibration_turns`) use `transcriber_fast`; later turns always use the full model
- Both models are pre-warmed at startup if both are configured

**(d) F5-TTS nfe_steps config:**
- `config/exhibit.yaml` adds `tts.nfe_steps: 8` (matches the previous hard-coded default; try 16/32 for higher fidelity at higher latency cost; try 4/6 for faster synthesis)
- `run_loop()` reads `nfe_steps` and passes it to `VoiceCloner(steps=nfe_steps)`
- `VoiceCloner` already accepted `steps` in its constructor; this just routes the config value to it

**(a) Streaming TTS — deferred:** F5-TTS-MLX's `sample()` call returns a complete waveform; true chunk-streaming would require forking the library. Deferred to a future patch against upstream.

**Tests:** 6 tests in `test_features_15_16_17.py` — all pass.

## Verification

**Audible verification:** deferred to exhibit Mac — this session ran without audio hardware.

**Latency numbers:** deferred to exhibit Mac. Estimated benefit of Feature 17:
- (b) Thread overlap: ~1–2 s saved when `engine.llm: true`; ~0 ms when `llm: false`
- (c) Fast Whisper model: depends on choice; `whisper-small` vs `large-v3-turbo` is ~3–5× faster on calibration turns (the turns where transcript quality is least critical)
- (d) `nfe_steps: 4` vs `nfe_steps: 8` is roughly 2× faster synthesis; quality impact is audible; document before deploying
